=== 1. Projektuebersicht ===

SyxBridge ist eine KI-gestuetzte Uebersetzungs-Engine fuer *Songs of Syx*-Mods.
Sie automatisiert den vollen Workflow:

    Scan -> Extract -> Translate -> Audit -> Polish -> Export

* Sprache: Node.js (v18+)
* Dashboard: Web-GUI auf http://localhost:3000
* Status: Alpha, Solo-Dev, taeglicher Einsatz
* Plugin-Architektur: v0.20 Phase 1 abgeschlossen (8/8 Hardcodes entkoppelt)
* PREFLIGHT-Analyse: automatischer DB-Health-Check vor jedem Sync
* Dual-Path-Copy (Native Mode): Steam-Workshop + AppData parallel


=== 2. Installation & erster Start ===

System-Voraussetzungen:
  - Windows 10/11 (oder Linux/macOS mit Node 18+)
  - 8 GB RAM minimum, 16 GB empfohlen
  - ~500 MB freier Speicherplatz (REVIEW BASE ohne node_modules)

Schritte:
  1. ZIP entpacken, z.B. nach C:\SyxBridge_REVIEW_BASE\
  2. Sicherstellen, dass Node.js v18 oder neuer installiert ist:
        node --version
  3. Im Projekt-Root ausfuehren:
        start.bat
     Der Launcher fuehrt bei Bedarf 'npm install' aus.
  4. .env konfigurieren (API-Keys) — beim ersten Start wird eine
     .env.example-Vorlage kopiert, ergaenze deine Provider-Keys.
  5. Dashboard oeffnet sich automatisch auf http://localhost:3000.


=== 3. Schnellstart: CLI-Modus ===

Fuer interaktive Steuerung:

    node core/index.js

Flags (typisch):
  --auto                 ohne Rueckfragen durchlaufen
  --mod <name>           nur Mod-Name bearbeiten (z.B. --mod V71)
  --langs de,en          Zielsprachen (Komma-getrennt)
  --provider groq,nvidia Provider-Reihenfolge
  --polish               Deep-Polish-Run erzwingen
  --dry                  kein Schreiben, nur Report
  --no-gui               ohne Dashboard-Start

Beispiel:

    npm run reconstruct -- --auto --mod V71 --langs de


=== 4. Schnellstart: GUI-Modus ===

Start.bat ohne Argumente startet das WEB-Dashboard auf Port 3000.

GUI bietet:
  - Live-Run-Controls (Start/Pause/Resume/Cancel)
  - Provider-Status + Live-ETA-Box
  - Heartbeat-Timer (alle 2s) und Sub-Phase-Tracking
  - DB-Suche ueber alle ~6.300 Eintraege
  - Revision-Restore (pro Eintrag)
  - Flag-Inspect + Manual-Override
  - Settings (Provider-Keys, Routing, Cost-Limits)


=== 5. Die Pipeline im Detail ===

Phase 1: Scan
  - Liest Workshop + AppData-Verzeichnisse
  - Klassifiziert Files via Adapter (SongsOfSyxAdapter)
  - Risiko-Score pro Mod-File (Static + Dynamic)

Phase 2: Extract
  - Zerlegt Strings via Plugin (SongsOfSyxPlugin)
  - Shield-Token-Format: [[N]] -> __SHLD_N__ (Schutz)
  - Marker-Validation (Tags, Placeholder, Variablen)

Phase 3: Translate
  - 9 Provider stehen bereit (siehe unten)
  - Risk-basiertes Routing: niedrige Risk -> schneller Provider
  - Cost-Klassen: Argos=10 (Fallback), Groq=4, NVIDIA=4, Ollama=1
  - Error-Handler: 429 -> eskalierender Cooldown (nicht mehr permanent disable)

Phase 4: Audit
  - Quality-Score pro Eintrag (translationCriticalCheck)
  - 3-Tier Accept/Reject (translationLooksSafe + assessWarnings)
  - BUG-FS-003 Doppelshielding fuer Non-LLM Provider

Phase 5: Polish
  - Multi-Provider A/B-Polishing (polish-arbiter)
  - Deep Polish Queue fuer mangelhafte Eintraege
  - Streaming-Writes mit subPhase='writing'

Phase 6: Export
  - Datei-Export mit Syntax-Validierung (exporter.js)
  - Dual-Path-Copy (Workshop + AppData, Native Mode)
  - Marker-Level-Validierung blockt defekte Writes (SHIELD_RESTORE_FAIL)

  - cache_provider (Provider-Health-Cache)

  Format: translations_YYYY-MM-DD_{tag}.tar.gz
  Tags:    before_<fix>  /  after_<fix>  /  session_<thema>  /  release
  Pfad:    core/archive/dbold/

Lifecycle:
  - Vor jedem groesseren Fix:  User nach Backup fragen
  - Nach jeder Session:        User nach Archivierung fragen (>5% Metrik-Drift)
  - Snapshot Reports nach:     core/archive/dbold/DB_TREND_REPORT.md
                               core/archive/dbold/DB_STATISTICS.md

=== 1. Project Overview ===

SyxBridge is an AI-powered translation engine for *Songs of Syx* mods.
It automates the full workflow:

    Scan -> Extract -> Translate -> Audit -> Polish -> Export

* Language: Node.js (v18+)
* Dashboard: Web-GUI on http://localhost:3000
* Status: Alpha, solo-dev, daily use
* Plugin architecture: v0.20 Phase 1 complete (8/8 hardcodes decoupled)
* PREFLIGHT analysis: automatic DB health check before every sync
* Dual-path copy (Native Mode): Steam Workshop + AppData in parallel


=== 2. Installation & First Launch ===

System requirements:
  - Windows 10/11 (or Linux/macOS with Node 18+)
  - 8 GB RAM minimum, 16 GB recommended
  - ~500 MB free disk space (REVIEW BASE without node_modules)

Steps:
  1. Unzip, e.g. to C:\SyxBridge_REVIEW_BASE\
  2. Make sure Node.js v18+ is installed:
        node --version
  3. From project root, run:
        start.bat
     The launcher runs 'npm install' if needed.
  4. Configure .env (API keys) — on first launch a .env.example
     template is copied; fill in your provider keys.
  5. Dashboard opens automatically on http://localhost:3000.


=== 3. Quickstart: CLI Mode ===

For interactive control:

    node core/index.js

Common flags:
  --auto                 non-interactive run-through
  --mod <name>           only this mod (e.g. --mod V71)
  --langs de,en          target languages (comma-separated)
  --provider groq,nvidia provider order
  --polish               force a deep-polish run
  --dry                  read-only, only emit a report
  --no-gui               do not start the dashboard

Example:

    npm run reconstruct -- --auto --mod V71 --langs de


=== 4. Quickstart: GUI Mode ===

start.bat without arguments launches the WEB dashboard on port 3000.

GUI provides:
  - Live run controls (Start/Pause/Resume/Cancel)
  - Provider status + live ETA box
  - Heartbeat timer (every 2s) and sub-phase tracking
  - DB search across all ~6,300 entries
  - Revision restore (per entry)
  - Flag inspect + manual override
  - Settings (provider keys, routing, cost limits)


=== 5. The Pipeline in Detail ===

Phase 1: Scan
  - Reads Workshop + AppData directories
  - Classifies files via adapter (SongsOfSyxAdapter)
  - Risk score per mod file (static + dynamic)

Phase 2: Extract
  - Decomposes strings via plugin (SongsOfSyxPlugin)
  - Shield token format: [[N]] -> __SHLD_N__ (protection)
  - Marker validation (tags, placeholders, variables)

Phase 3: Translate
  - 9 providers available (see below)
  - Risk-based routing: low risk -> fast provider
  - Cost classes: Argos=10 (fallback), Groq=4, NVIDIA=4, Ollama=1
  - Error handler: 429 -> escalating cooldown (no longer permanent disable)

Phase 4: Audit
  - Quality score per entry (translationCriticalCheck)
  - 3-tier Accept/Reject (translationLooksSafe + assessWarnings)
  - BUG-FS-003 double-shielding for non-LLM providers

Phase 5: Polish
  - Multi-provider A/B polishing (polish-arbiter)
  - Deep polish queue for subpar entries
  - Streaming writes with subPhase='writing'

Phase 6: Export
  - File export with syntax validation (exporter.js)
  - Dual-path copy (Workshop + AppData, Native Mode)
  - Marker-level validation blocks defective writes (SHIELD_RESTORE_FAIL)
