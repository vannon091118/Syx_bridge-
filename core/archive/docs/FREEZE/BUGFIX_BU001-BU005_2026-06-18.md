## BU-003 — persistConfigToEnv Clobbert User-Env-Vars
## ══════════════════════════════════════════

### Symptom
`persistConfigToEnv(config)` in `core/src/config-runtime.js` rief einmal `fs.promises.writeFile(envPath, ...)` mit einem hartkodierten 25-Key-Array auf. Resultat: bei jedem GUI-Save oder CLI-Wizard-Lauf wurden alle vom User manuell hinzugefügten Env-Vars (`LOG_LEVEL`, custom OpenAI-compatible Endpoints, NMT-Settings, etc.) aus `.env` gelöscht.

### Root Cause
Ursprüngliche Implementierung kannte nur "die 25 Keys, die SyxBridge verwaltet". User-Customizations fielen durch die Maschen.

### Fix
Refactor intern auf existierende `persistSingleEnvVar()`-Funktion:
1. Key-Liste als Modul-Konstante `PERSISTED_KEYS` extrahiert (jeder Eintrag ist `[key, extractor(config)]` für deklarative Lesbarkeit).
2. `persistConfigToEnv` loopt jetzt über `PERSISTED_KEYS` und ruft pro Key `persistSingleEnvVar` auf.
3. **`persistSingleEnvVar` preserviert alle unbekannten Zeilen, Kommentare und Leerzeilen** in `.env`. Der Fix nutzt diese Eigenschaft bewusst.
4. Polish-Pass: try/catch pro Key — single-failure darf andere Keys nicht killen, Fehler werden gesammelt + als `[WARN]` geloggt.

### Tradeoff-Dokumentation
- **Vorher:** Eine atomare `writeFile` (alle 25 Keys oder 0).
- **Nachher:** 25 sequentielle Atomar-Writes — User-Vars bleiben erhalten, aber Loop-Abbruch (selten) kann Partial-State hinterlassen. Mitigation: `process.env` wird von `persistSingleEnvVar` synchron aktualisiert, daher reflektiert `applyEnvToConfig()` beim nächsten Start den Live-CONFIG (auch wenn .env veraltete Zeilen hat).

### Verifikation
- `node --check core/src/config-runtime.js` → OK ✅
- `e2e_p5_sprachauswahl.js` testet `persistSingleEnvVar` direkt — Test bleibt grün (Funktionsverhalten unverändert).
- Bestehende `persistConfig`-Aufrufer (gui-handlers.js:607, ConfigRuntime.configure() in config-runtime.js:762) profitieren ohne Aufrufer-Änderung.

## BU-004 — Backup Race Condition ohne Lock
## ══════════════════════════════════════════

### Symptom
`translateMod()` in `core/src/runtime-ops.js:124-159`: zwei parallele CLI-Runs konnten gleichzeitig erkennen, dass `backupPath` fehlt → beide riefen `fsp.cp(modDir, backupPath, {recursive:true})` auf → EEXIST-Catch akzeptierte das Resultat, **aber das Backup war möglicherweise halbfertig oder korrupt geschrieben**.

### Root Cause
- `fs.existsSync` ist TOCTOU — zwischen Check und Write kann ein anderer Run den Pfad anlegen oder löschen.
- `fsp.cp` bietet keine mkdir+cp-Atomicity-Garantie.
- Der EEXIST-Catch reagierte NUR auf einen spezifischen OS-Fehler NACH potentiellem Datenverlust.

### Fix-Strategie (Thinker-with-Files-Gemini analysiert)
**File-based Mutex Lock** — keine externe Dependency (kein `proper-lockfile`):

1. **`acquireBackupLock(lockPath, lockLabel)`** — atomare Acquire via `fs.writeFileSync(path, payload, {flag:'wx'})`. `wx` = O_CREAT|O_EXCL auf POSIX und NtCreateFile mit FILE_SHARE_NONE auf Windows. Wartet mit 500ms Polling bis 30s Timeout.
2. **Stale-Lock-Erkennung:** Wenn Lock gehalten wird, lesen wir `{pid, startedAt}` und prüfen `process.kill(pid, 0)`:
   - `ESRCH` → PID tot → Lock sofort übernehmen
   - `EPERM` → lebt, aber kein Signal-Recht → als alive behandeln
   - 5-Min-Timestamp-Fallback → falls PID recycelt wurde
3. **`releaseBackupLock(lockPath)`** — wrapped in `finally` des Aufrufers, sodass ein `cp()`-Throw den Lock nicht leakt. `unlinkSync` ist silent auf ENOENT.
4. **Double-Checked Locking:** Nach Lock-Acquire wird `shouldBackup` ein zweites Mal geprüft (im Fall dass ein paralleler Run inzwischen den Backup erstellt hat — wir akzeptieren dann dessen Backup).

### Verhalten bei Lock-Timeout
`throw new Error("Backup-Lock konnte ... nicht erworben werden. Abbruch, damit kein Run ohne verifiziertes Backup startet.")`. Im Native Mode darf kein Run ohne verifiziertes Backup starten → Abbruch statt „best-effort" ist der korrekte Default.

### Verifikation
- `node --check core/src/runtime-ops.js` → OK ✅
- Lock-Pfad: `BACKUP_ROOT/.lock_<backupId>` (im selben Verzeichnis wie Backup).
- 60 Polling-Iterationen × 500ms = Max 30s Wartezeit pro Mod.
- Cleanup-Pfad (`fsp.rm(backupPath)` für orphan-backups) liegt jetzt INNERHALB der Lock-Critical-Section → Race-frei.

## BU-005 — cleanup_argos_stale.js ignoriert WAL
## ══════════════════════════════════════════

### Symptom
`scripts/cleanup_argos_stale.js` nutzte `sql.js` (pure-JS Implementation). sql.js **liest nur die Haupt-DB-Datei** und ignoriert komplett die WAL-Datei (`translations.db-wal`). Folge:
1. **Vor dem Cleanup:** Noch nicht gemergte Changes aus einem parallelen SyxBridge-Run wurden ignoriert → Skript sieht veralteten Snapshot.
2. **Nach dem Cleanup:** `db.export()` schreibt nur die Haupt-DB → die WAL des parallelen Run wird beim nächsten Öffnen inkonsistent re-applied → potentiell korrupte Daten oder SQLITE_CANTOPEN.

### Root Cause
sql.js ist die einzige pure-JS-SQLite-Alternative, aber sie unterstützt kein WAL. Native sqlite3 (via `sqlite3` npm) kann WAL lesen + `PRAGMA wal_checkpoint(TRUNCATE)` ausführen.

### Fix — Komplett-Rewrite
1. **Engine-Switch:** `initSqlJs` + `fs.readFileSync(DB_PATH)`-Buffer-Pattern ersetzt durch `core/src/db.js`-Import (Promise-basierter Wrapper um native sqlite3).
2. **WAL-Checkpoint PRE:** `PRAGMA wal_checkpoint(TRUNCATE)` vor dem ersten DB-Read — mergt pending writers und liefert konsistenten Snapshot. Output-Logging: `[OK] WAL-Checkpoint (pre): busy=...`.
3. **WAL-Checkpoint POST:** Nach den DELETEs — schreibt die Änderungen in die Haupt-DB und truncatiert die WAL-Datei. Resultat: nach Skript-Ende existiert nur EIN konsistenter DB-Zustand auf der Platte.
4. **30s-Timeout pro Operation** (`runWithTimeout`) — verhindert unendliches Hängen bei `SQLITE_BUSY` durch parallelen Lock-Holder.
5. **Defensiver Close:** `if (db.db() && typeof db.db().close === 'function')` vor dem Finalize, damit ein misslungener Init nicht den Process-Exit crasht.

### Verifikation
- `node --check core/scripts/cleanup_argos_stale.js` → OK ✅
- Verhalten bei leerer DB: Skip-Path mit `No entries found.` (unverändert).
- Verhalten bei „WAL nicht leer": `postCheckpoint.checkpointed_frames > 0` würde jetzt tatsächlich die WAL mergen (vorher: stumm ignoriert).
