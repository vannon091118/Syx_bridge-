## 1. Grundprinzipien (Vom User definiert)

> **P1: RESTORE ist die einzige Funktion im System die vollfunktionstüchtig vorhanden ist aber nie genutzt werden muss.**
> RESTORE existiert als ultimatives Safety-Net — aber der primäre Übersetzungs-Pfad MUSS so zuverlässig sein, dass RESTORE nie greift. Wenn RESTORE aktiviert wird, ist das ein Bug, kein Feature.

> **P2: Qualität ist Polish und optional solange das Ergebnis qualitativ als "lesbar" gilt.**
> Deep-Polish, Proofreading, Terminology-Checks = Nice-to-have, nicht Gatekeeper. Solange die Übersetzung LESBAR ist (Score ≥ 50), akzeptieren wir sie. Score < 50 = nicht lesbar → MUSS gefixt werden. Aber: Eine schlechte Übersetzung ist besser als keine Übersetzung.

> **P3: Jedes UseCase-Szenario wird zu einem Qualitäts-Unterschied führen, niemals aber zu einem Ergebnis-Unterschied. Wir liefern IMMER!**
> Verschiedene Provider/Approaches → verschiedene Qualität (ok). ABER: Jeder Pfad MUSS ein valides Ergebnis liefern. Nie leer, nie error, nie korrupt. "Zero Result Difference" = immer eine Übersetzung, nie ein Fehler.

### 🔴 Fehlerquelle #1: Watermark-Akkumulation (P0 — Release-Blocker)

**Symptom:** 423 übersetzte Strings werden als "stale" behandelt und nie wieder re-translated.

**Root Cause:** `applyTranslations()` in `text-core.js:514-519` injiziert ZWSP/ZWNJ nach dem ersten Wort JEDES übersetzten Strings. Beim nächsten Run:
1. Source-Text auf Disk hat bereits Watermarks (aus vorherigem Run)
2. DB-Vergleich: `source_text === translation` (beide markiert)
3. `classifyNativeDecision()` → `isLikelyTargetLanguageText(source)` = TRUE (es ist Deutsch mit Watermarks)
4. Ergebnis: `{ reuse: true, reason: 'already_target_lang' }` → stale

**Akumulation:** Jeder Re-Run fügt NEUE Watermarks hinzu. Strings korumpieren über die Zeit.

**Beweis:** Alle 20 sampled stale native_runtime-Einträge mit deutschen Sätzen hatten `HAS_WATERMARK`-Status. Beispiele:
- *"Wir sollten auf jeden Kampf vorbereitet sein..."* — klar Deutsch, kein Proper Noun
- *"{RACE_CITY} ist das Paradies!"* — vollständiger Satz mit Placeholder

**Lösung:** Watermark NUR in Output-Dateien injizieren, NICHT in den DB-Source. Oder: Watermark aus Source extrahieren VOR Classification.

---

### 🔴 Fehlerquelle #2: shouldTranslate() False Positives (P0)

**Symptom:** 23 LOW_SCORE_FLAGGED (Score 20-25) + 5 STRUCTURAL_TRUNCATED = Config-Syntax-Fragmente die nie hätten übersetzt werden sollen.

**Root Cause:** `shouldTranslate()` in `text-core.js:117` akzeptiert SoS-Konfigurations-Syntax:
- `},\n\tHEAL1: {\n\t\tNAME:` — JSON-ähnliche Key-Value-Paare
- `},\r\nview.menu.ScKeys: {\r\n\tnameBig:` — Java-Paketnotation
- `ARMY_NAMES: [\r\n\t` — Array-Syntax

Die 23 Einträge wurden an Provider (polish_single, google_free, groq) geschickt und auf `}` gekürzt.

**Beweis:** CAMEL_CASE entries = 6.462 (68% der DB), 86% stale. Die 23 LOW_SCORE sind strukturelle Code-Fragmente.

**Zusätzliche Regex-Regeln für shouldTranslate():**
- `/^[\s},]*[A-Z0-9_]+:\s*\{?[\sA-Z0-9_:]*$/` — Blockt Config-Blöcke
- `/^[A-Z_]+:\s*\[[\s\r\n]*$/` — Blockt Array-Syntax
- `/^[a-z]+\.[a-z]+\.[a-z]+\.[A-Z]/` — Bereits vorhanden (Java-Pfade)

## 4. RESTORE-Philosophie (Code-Garantien)

### Was "RESTORE nie nutzen" konkret bedeutet:

1. **Zero-Destruction-Guarantee:**
   Jeder Provider-Output der Struktur-Checks (`translationCriticalCheck`, `placeholdersValid`) verfehlt, MUSS auf den Original-Source zurückfallen. Niemals korrupte Daten in die DB schreiben.

2. **Never-Empty:**
   Leere Rückgabe, API-Timeout, 500er-Fehler → File nicht anrühren (Fallback = Original). `applyTranslations()` modifiziert das File NUR wenn das Ergebnis zu 100% verifiziert ist.

3. **Fail-Safe Persistence:**
   Schlechter Text (Score < 50) wird als `requires_deep_polish` asynchron persistiert, OHNE das aktuelle Spiel zu brechen. Die Übersetzung ist lesbar — sie ist nur nicht gut.

4. **Watermark-Safety:**
   Watermarks werden NUR in Output-Dateien injiziert, NIE in den DB-Source. Der DB-Source bleibt "sauber" für Re-Runs und Classification.

## 5. Qualitäts-Definition

### Was "lesbar" bedeutet:

| Score | Qualitäts-Level | Aktion |
|-------|----------------|--------|
| 90-95 | Exzellent | Akzeptiert, optionaler Deep-Polish |
| 70-89 | Gut | Akzeptiert, kein Eingriff nötig |
| 50-69 | Akzeptabel | Akzeptiert, Deep-Polish empfohlen |
| 30-49 | Schlecht | Akzeptiert NUR wenn keine Alternative, Flag für Review |
| 0-29 | Nicht lesbar | MUSS re-translated werden (P1 Recovery) |

### Qualitäts-Unterschied vs. Ergebnis-Unterschied:

- **Qualitäts-Unterschied:** Groq liefert bessere Übersetzung als google_free → ok
- **Ergebnis-Unterschied:** Groq liefert Übersetzung, google_free liefert nichts → NICHT ok

Jeder Provider MUSS ein valides Ergebnis liefern. Wenn ein Provider versagt, greift der nächste in der Routing-Kette. Erst wenn ALLE Provider versagen, greift der native_fallback (Original-Text behalten). RESTORE greift NUR wenn das File bereits korrupt ist — das darf nie passieren.
