C4 — Argos ETIMEDOUT Fix: exponentieller Backoff stoppt 23× GUI-Polling-Spam

Nach der GUI-Polish (C3), dem .gitignore-Konflikt (C2) und den Dev-Tools (C1), nach B1s Engine-Bugfixes, B2s Infrastruktur-Härtung, B3s Doku-Audits, B4s Release-Vorbereitung und B5s Narrativ-Etablierung, meldete sich ein altes Problem zurück: der Argos-Timeout-Spam. 23 mal hintereinander stand im Log die Meldung getAvailableArgosLanguages fehlgeschlagen: spawnSync python ETIMEDOUT. 23 Warnungen in 4 Minuten. Das ist kein Fehler mehr, das ist ein Retry-Loop ohne Exit-Strategie der das Log mit nutzlosen Einträgen zumüllt und dabei CPU-Zyklen für sinnlose Python-Prozesse verschwendet. Die Ursache: eine fehlende Cooldown-Logik in Kombination mit einem GUI-Polling-Intervall von 10 Sekunden und einem Python-Spawn-Timeout von 5 Sekunden.

B1 hatte die Core Engine mit sechs Bugfixes stabilisiert, aber der Argos-Code liegt ausserhalb der Engine in den Scripts und GUI-Polling-Routinen. B2 die Infrastruktur mit verify_commit_msg.js, check_vendor_drift.js und dem Dependency-Cleanup gehärtet, aber der Argos-Code war kein Teil dieser Härtung. B3 die Doku mit dem CODE_VS_DOCS_AUDIT durchforstet der 40 Abweichungen zwischen Doku und Code aufdeckte — darunter auch die ETIMEDOUT-Problematik die bereits im CHANGELOG dokumentiert war, aber nie gefixt wurde. B4 das Release mit FREEZE_INDEX und MASTER_DOC vorbereitet, B5 die Narrative etabliert. C1 die Dev-Tools mit git rm --cached entfernt, C2 den .gitignore-Drei-Wege-Konflikt gelöst, C3 die GUI auf v0.20.0 poliert. Jetzt C4 — der Spam der das Log unlesbar machte.

Die Problem-Chain im Detail: isArgosInstalled() cached beim ersten Aufruf das Ergebnis true, weil Python zu dem Zeitpunkt läuft. Danach startet das GUI-Polling alle 10 Sekunden den Callback get-argos-languages der getModelStatus() aufruft. getModelStatus() ruft getAvailableArgosLanguages() auf, das via spawnSync python ein Python-Script startet um die verfügbaren Argos-Sprachen abzufragen. spawnSync hat einen Timeout von 5000 Millisekunden — danach wirft es ETIMEDOUT. 6 Polls pro Minute × 5 Sekunden Timeout = 30 Sekunden Wartezeit pro Minute. Nach 4 Minuten sind 23 Fehler im Log, der Rechner hat 115 Sekunden mit Warten auf Python verbracht, und kein einziger Versuch war erfolgreich.

Die Lösung implementiert drei Mechanismen. Erstens: ein exponentieller Backoff mit der Variablen _argosCooldownMs die bei 30000 Millisekunden startet und sich nach jedem Fehler verdoppelt: 30s, 60s, 120s, 240s, bis zum Maximum von 300s. Zweitens: isArgosOnCooldown() prüft ob die aktuelle Zeit minus _lastArgosFailureTime kleiner ist als _argosCooldownMs — wenn der Cooldown abgelaufen ist, wird _argosCooldownMs auf 30000 zurückgesetzt und false zurückgegeben. Drittens: markArgosFailure() setzt _lastArgosFailureTime auf Date.now(), verdoppelt _argosCooldownMs bis zum Maximum von 300000 Millisekunden, setzt argosInstalledCache auf null und pythonCache auf null. Die Caches werden invalidiert damit der nächste Versuch Python neu entdeckt — falls das Problem nur ein temporärer Connection-Timeout war, funktioniert es nach dem Cooldown wieder. Falls Python dauerhaft weg ist, bleibt der Cooldown aktiv und das System lernt schnell dass Argos nicht verfügbar ist. Während des Cooldowns geben getAvailableArgosLanguages() und checkArgosLanguages() leere Objekte zurück — keine Logs, keine Prozesse, kein CPU-Overhead. Der Fix ist live auf Disk.
