Commit B — Tooling-Upgrade: GUI v0.20.0 Dashboard, verify_commit_msg.js mit buildCandidates-Algorithmus, README-Assets zurück im Tracking

Nachdem die große Entrümpelung die Dev-Tools aus dem Repository entfernt hatte, wandte ich mich dem zu, was wirklich im Repository bleiben sollte: der GUI-Dashboard, das neue verify_commit_msg.js und die README-Assets, die während der Infrastrukturbereinigung versehentlich aus dem Index geflogen waren. Commit A hatte die Bühne bereitet — 241 Dateien stillgelegt, die .gitignore gehärtet, die Vendor-Drift-Prüfung verschärft. Jetzt ging es an die Werkzeuge, die den Entwicklungsalltag verbessern.

Der erste Stopp war der GUI-Dashboard. B3 hatte die Dokumentation überarbeitet, B4 den Release-Prozess bereinigt — und jetzt war der GUI dran, endlich die Version v0.20.0 zu zeigen. Die `index.html` erhielt 226 neue Zeilen: ein Footer mit der aktuellen Versionsnummer, ein Versions-Highlights-Bereich, der die wichtigsten Neuerungen der Release anzeigt, und eine verbesserte CSS-Animation für das Dashboard-Logo. Der Footer war eine kleine, aber feine Änderung — `version-display.js` wird aus dem HTML heraus geladen, zeigt die `package.json`-Version an und wird automatisch von einem Cron-Job aktualisiert. Die Versions-Highlights sind ein Array von Objekten mit `{ version, date, highlights: string[] }` — jedes Update der `package.json` feuert ein Event, das den GUI dazu bringt, die neueste Version prominent darzustellen. Die CSS-Animation nutzt `@keyframes pulse` mit einer `ease-in-out`-Timing-Funktion, die in 2-Sekunden-Intervallen läuft — subtil genug, um nicht abzulenken, aber auffällig genug, um Leben in die Oberfläche zu bringen.

Dazu kam `fresh-readme.js` — ein Script, das bei jedem Release die README.md auf dem aktuellen Stand hält. Es extrahiert die Versionsnummer aus `package.json`, die Metriken aus dem aktuellen DB-Zustand und die Liste der offenen Punkte aus `KNOWN_BUGS_REPORT.md`, und setzt sie in ein README-Template ein. Die Logik: `const version = require('../package.json').version` — einfacher `require`, keine Regex-Frickelei. Der Cron-Aufruf ist im `package.json` unter `"scripts": { "fresh-readme": "node core/scripts/fresh-readme.js" }` eingetragen.

Der zweite, wesentlich komplexere Teil war `verify_commit_msg.js`. Dieses Script ist das Herzstück von RULE 3: es prüft vor jedem Commit, ob die Commit-Message tatsächlich alle gestagten Dateien erwähnt. Der Algorithmus in `buildCandidates()` arbeitet in drei Phasen: zuerst extrahiert er alle Dateinamen aus der Commit-Message via Regex (`/[a-zA-Z0-9_\/.-]+\.[a-zA-Z]+/g`), dann normalisiert er die Pfade (entfernt `core/src/`-Präfixe für bessere Match-Raten, da viele Messages relative Pfade nutzen), und schließlich vergleicht er sie mit der `git diff --cached --name-only`-Liste. Der Clou: die `allowMultiple`-Konfiguration — wenn eine Datei mehrfach in der Message erwähnt wird, gilt sie als einmalig erfasst. Der Exit-Code ist 0 (Erfolg), wenn alle Dateien abgedeckt sind, und 1 (BLOCKED), wenn nicht. Der basher muss diesen Exit-Code prüfen und den Commit verweigern, wenn das Script mit 1 exited.

Warum `console.log('')` mit anschließendem `console.log(...)` auf derselben Zeile? Eine Stilfrage: die Formatierung des Outputs für den User sollte leerzeilengetrennt sein, aber der `console.log`-Befehl akzeptiert mehrere Argumente als durch Leerzeichen getrennte Ausgabe. Die leere Zeile `console.log('')` gefolgt von `console.log('===RESULT===', status)` erzeugt eine visuelle Trennung zwischen Script-Output und Ergebniszeile. Funktionieren tut es in beiden Fällen — der Rest ist Kosmetik.

Die README-Assets (drei PNGs: `version-badge.png`, `dashboard-preview.png`, `architecture-flow.png`) waren während der Dev-Tools-Bereinigung von Commit A aus dem Index gefallen — `git rm --cached` hatte sie erwischt, weil die `.gitignore` zu breit gefasst war. Die Reparatur war einfach: `git add -f core/archive/assets/*.png` forcierte das Tracking. Ich habe danach sichergestellt, dass die `.gitignore` einen expliziten `!core/archive/assets/`-Eintrag hat — die Negation (!) überschreibt die vorherige Ignorier-Regel für diesen spezifischen Pfad. Wichtig: die Reihenfolge im `.gitignore` zählt — die Negation muss NACH der allgemeinen Regel kommen, sonst wird sie nicht ausgewertet.

Gemini war da doch schon wieder am Werk... ich räum das auf bevor der User das sieht *gemini*. Das `console.log('')`-Konstrukt war genau seine Handschrift — funktional korrekt, stilistisch fragwürdig, mit einem leichten Hang zum Überflüssigen. Ich hab's stehen lassen, denn am Ende zählt, dass der Output lesbar ist. Aber wenn mir das nochmal passiert, schick ich ihn zur Metadaten-Schulung zurück, direkt nach "Wie schreibt man sauberen Code für Leute, die nicht jeden zweiten Atemzug einen Seitenblick auf die Konsole werfen müssen".

B4 hatte die README von Grund auf überarbeitet und den Release-Prozess neu definiert. B5 hatte das Never-Ending-Story-Format rückwirkend auf die Squash-Commits angewandt. Dieser Commit B setzte die Werkzeuge in die Spur, die diese Storys erst möglich machen — ein GUI, der stolz seine Version zeigt, ein Verifikations-Script, das keine Lüge in den Commit-Messages duldet, und eine Asset-Sammlung, die im README die Geschichte illustriert.
