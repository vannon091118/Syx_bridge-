## PHASE 4: DATENFLUSS

**Primärer Übersetzungs-Datenfluss**

```
Steam Workshop .txt-Dateien
    → collectTextFiles() → readFileJob()
    → extractReplacements()                     [Regex über alle "..." ohne Key]
    → allTexts[] = { source, relativePath, ... }

allTexts
    → ensureTranslations()
        → enrichWithContext()                   [Glossar-Lookup, Risk-Score]
        → getCachedTranslations()               [⚠️ N+1 seriell]
        → classifyNativeDecision()              [Proper Noun / Already German?]
        → missing = nicht gecachte Texte
        → while loop: translateBatchWithRouting()
            → dispatcher.runRoute()
                → translateBatch(items, provider)
                    → buildBatchPromptForCurrentConfig()
                        → shieldPlaceholders()  [Round 1: <x> → [[0]]]
                        → Prompt-Text mit [[0]] etc.
                    → inline: protectPlaceholders()
                        → shieldPlaceholders()  [Round 2: gleicher Input, gleiche Map]
                    → call[Provider]Batch(entries)
                        → API-Call mit Round-1-Prompt
                        → parseBatchResponseWithMaps()
                            → parseBatchResponse()  [shieldMaps-Parameter ignoriert]
                            → returned Strings mit [[0]] tokens drin
                    → rawTranslations[i] hat [[0]] tokens
                    → restoreAndValidateTranslation(source, rawTranslation, Round2.placeholders)
                        → restorePlaceholders()     [[[0]] → <x>]
                        → translationLooksSafe()
                    → falls invalid: Fallback auf Original-Text (Quelle)
                → result.translations[]
        → saveTranslation()                     [UPSERT in DB]
        → learnGlossary()                       [Glossar-Update]
        
        [Optional GRAMMAR_CHECK-Phase]
        → flagPotentialErrors()                 [Audit-LLM: bool[]]
        → fixGrammarBatch()                     [Polish-LLM]
            → buildProofreadPrompt()            [shieldMaps korrekt genutzt]
            → executeStageRequest()
                → parseBatchResponseWithMaps()
                    → shieldMaps HIER genutzt   [Restore im parseRaw()]
        → saveTranslation(..., polishLevel=2)

translations (Map)
    → writeTranslatedFile()
        → applyTranslations(content, replacements, translations)
            → replacements ignoriert, Datei erneut per Regex gescannt
        → fs.writeFile(outPath)
    → markProcessed()                           [processed_files UPSERT]

Datei-Output → PATCH_ROOT/[ModName]_German/
    → bundleBridgeCore() → GAME_MOD_ROOT/BridgeCore/
    → syncLauncherSettings() → LauncherSettings.txt
```

**Wo werden Daten still verworfen:**

1. **`parseBatchResponse` ignoriert `shieldMaps`** (options.shieldMaps wird übergeben, nie gelesen). Round-1-ShieldMaps von `buildBatchPromptForCurrentConfig` gehen verloren. Funktioniert nur, weil Round 2 (inline `protectPlaceholders`) identische Maps erzeugt.

2. **`applyTranslations(content, replacements, translations)`** — Parameter `replacements` wird nie gelesen. Die Funktion re-scannt `content` mit einem eigenen Regex. Alle vorberechneten `replacements` aus `readFileJob` sind bei Schreibzeit wertlos.

3. **`translateBatch` Fallback bei invalider Translation:** `return item.source` — das Original wird verwendet und als Translation gespeichert, mit `flagReason: ''` wenn der Validation-Pfad schlägt. Erst der N+1-Weg via `inferFlagReason` setzt korrekte Flags, aber `restoreAndValidateTranslation`-Fallback setzt keinen separaten Flag-Grund.

4. **Fehlerbehandlung in `ensureTranslations` catch-Block:** Fehlgeschlagene Batches werden mit `provider: 'native_fallback', flagReason: 'all_routes_failed', qualityScore: 20` in der DB gespeichert. Das Original wird als Translation verwendet. Das ist korrekt dokumentiert, aber `console.error` geht in `log.txt` — bei `--gui` wird kein extra Event gefeuert.

**Fehlerbehandlungspfade:**

- `callXxxBatch` 429/401: Key-Rotation + Retry → wenn alle Keys erschöpft: Exception nach oben
- `dispatcher.runRoute`: iteriert routePlan, bei Fehler `routingEngine.handleFailure()` → Cooldown → nächste Route
- Alle Routen fehlgeschlagen: `throw new Error("Alle ... Routen fehlgeschlagen")` → `ensureTranslations` catch → Original gespeichert
- `callArgosBatch` bei Python-Fehler: `return texts.map(t => t)` → Silent Fallback auf Originale, kein Flag
- `callGoogleTranslateFree` Einzel-Fehler: `results.push(text)` → Original ohne Flag

## PHASE 5: CLAIM-VERIFIKATION

**CLAIM: "Platzhalter wie `__VAR0__` überleben die Übersetzung"**
→ **STATUS: TEILWEISE VERIFIZIERT**
→ **BEWEIS:** `shieldPlaceholders` regex deckt `<[^>]+>|__VAR\d+__|{[^}]+}|\$[A-Za-z0-9_]+|%[^%\s]+%` ab. Restoration passiert via `restoreAndValidateTranslation`. Problem: `parseBatchResponse` in `text-core.js` strippe `replace(/^"|"$/g, '')` — wenn ein LLM die Übersetzung in Anführungszeichen wrapped (`"[[0]] Hallo"`), würde das führende `"` korrekt entfernt. Aber wenn ein Ergebnis mit `"` endet aus einem legitimen Grund (Zitat), wird es gekürzt. Placeholder-Protection selbst: verifiziert funktionsfähig.

**CLAIM: "Bekannte Wörter kommen aus dem Glossar direkt ohne API-Aufruf"**
→ **STATUS: VERIFIZIERT**
→ **BEWEIS:** `classifyNativeDecision()` in `translation-runtime.js` prüft `glossaryMap.get(source)` und setzt `reuse: true, reason: 'glossary_exact'` ohne API-Call. Cache-Hit-Pfad ebenfalls via `getCachedTranslations` vor API-Calls.

**CLAIM: "Qualitätsprüfung läuft über einen separaten Schritt"**
→ **STATUS: VERIFIZIERT**
→ **BEWEIS:** Nach Batch-Translation in `ensureTranslations` läuft eine explizite `if (config.GRAMMAR_CHECK)` QA-Phase mit `flagPotentialErrors()` + `fixGrammarBatch()`. Ist standardmäßig DEAKTIVIERT (`GRAMMAR_CHECK=true` muss explizit gesetzt sein).

**CLAIM: "Mehrere Keys pro Provider werden automatisch rotiert"**
→ **STATUS: VERIFIZIERT**
→ **BEWEIS:** `ConfigRuntime.rotateApiKey(provider)` inkrementiert `KEY_INDICES[provider]`, `getApiKey` verwendet `keys[index % keys.length]`. Rotation wird bei 429/401 in allen `callXxxBatch`-Funktionen ausgelöst.

**CLAIM: "CPU und RAM Auslastung siehst du live im Dashboard"**
→ **STATUS: TEILWEISE FALSCH**
→ **BEWEIS:** RAM korrekt via `os.freemem()/os.totalmem()`. CPU: `broadcastStats` ist Arrow-Funktion in `main()`, nutzt `this._cpuPrevTotal`. In CommonJS-Sloppy-Mode persistiert `this` auf `global` zwischen Interval-Calls — das **zufällig** funktioniert, ist aber kein stabiles Design. Ohne vorherigen State gibt der erste Call immer `cpuPct = 0`. Gemessen wird CPU-Zeit-Delta über OS-Ticks — kein echter `loadavg`-Ersatz.

**CLAIM: "Texte die sich nicht verändert haben werden nicht nochmal übersetzt"**
→ **STATUS: VERIFIZIERT**
→ **BEWEIS:** `shouldSkipFile()` prüft Hash + mtime gegen `processed_files`-Tabelle. `getCachedTranslations()` cached auf `source_text`-Ebene. Beide Ebenen funktionieren korrekt.

**CLAIM: "Das Tool findet deine Mod-Ordner automatisch über den Steam Workshop Pfad"**
→ **STATUS: TEILWEISE — Windows-Only**
→ **BEWEIS:** `CONFIG.MOD_ROOT` default ist `C:\Program Files (x86)\Steam\...` (Hardcoded Windows-Pfad). `SETTINGS_PATH` in `sos-runtime.js` nutzt `process.env.APPDATA` → undefined auf Linux/Mac → Pfad wird zu `/songsofsyx/settings/LauncherSettings.txt`. Guard `if (!fs.existsSync(SETTINGS_PATH)) return []` fängt das still ab.

## PHASE 6: AUFFÄLLIGKEITEN

### KRITISCH

**C1 — N+1 Serieller DB-Loop in `getCachedTranslations`**
Datei: `src/translation-runtime.js`, Funktion `getCachedTranslations`, ca. Zeile 659–670.
```js
for (const entry of entries) {  // N Iterationen
    const row = sourceHash
        ? await dbGet('SELECT ... WHERE target_lang = ? AND (source_text = ? OR source_hash = ?) ...')
        : await dbGet('SELECT ... WHERE source_text = ? ...');
}
```
Für jeden einzelnen Entry ein separater `await dbGet`-Aufruf, seriell geblockt. Bei einem Mod mit 500 eindeutigen Strings = 500 sequenzielle SQLite-Queries. Korrekt wäre ein einzelnes `WHERE source_text IN (...)` mit `dbAll`.

**C2 — `this._cpuPrevTotal` in Arrow-Funktion: Globaler Namespace-Pollutant**
Datei: `index.js`, Funktion `broadcastStats`, ca. Zeile 556–571.
```js
const broadcastStats = () => {  // Arrow: this = outer this
    if (this._cpuPrevTotal != null && ...) { ... }  // this = global in sloppy mode
    this._cpuPrevTotal = total;  // Setzt global._cpuPrevTotal
};
```
Im Strict-Mode: `TypeError: Cannot set properties of undefined`. In Non-Strict: stille globale Namespace-Verschmutzung. Die CPU-Berechnung ist per Design gebrochen weil der erste Call immer `cpuPct=0` liefert.

**C3 — Race Condition: `guiIdleResolver` vor Deklaration referenziert**
Datei: `index.js`, Zeile 545 (`on('idle')` Handler) vs. Zeile 554 (Deklaration) vs. Zeile 592 (Zuweisung).
```js
global.guiServer.on('idle', async () => {
    if (guiIdleResolver) await guiIdleResolver();  // Zeile 545
});
// ...
let guiIdleResolver = null;  // Zeile 554 — TDZ-sicher, aber...
// ...
return new Promise((resolve) => {
    guiIdleResolver = async () => { ... resolve(); };  // Zeile 592
});
```
Die `guiIdleResolver`-Zuweisung (Zeile 592) passiert NACH dem Start des GUI-Servers (Zeile ~500). Wenn `idle` vor Zeile 592 feuert (Browser-Tab bereits geschlossen während Start-Sequenz), bleibt `guiIdleResolver === null`, der Handler tut nichts, und die Promise resolvet nie → Prozess hängt.

**C4 — Menü-Option `'single'` und `'workshop'` sind Dead UI**
Datei: `index.js` + `src/ui.js`.
- `mainMenu()` in `ui.js` bietet `{ value: 'single' }` und `{ value: 'workshop' }` an.
- `'single'`-Handler ist dreifach auskommentiert; `startSingleManual` ist nie definiert. Selektion → `menu.action === 'single'` matcht auf keinen `else if` → Loop-Iteration ohne Aktion.
- `'workshop'`-Handler fehlt vollständig in der while-Schleife. Gleiche Konsequenz: stille No-Op.

### MITTEL

**M1 — Doppeltes Shielding in `translateBatch`**
Datei: `src/translation-runtime.js`, Funktion `translateBatch`, ca. Zeile 610–635.
```js

restoreAndValidateTranslation(item.source, translation, item.placeholders)
```
Round-1-ShieldMaps werden erzeugt, übergeben, und stillschweigend ignoriert. Das System funktioniert nur weil `protectPlaceholders` deterministisch ist. Falls sich die Inputs jemals unterscheiden, entsteht stille Korruption ohne Fehlermeldung.

**M2 — `parseBatchResponse` strippe führende/schließende Anführungszeichen**
Datei: `src/text-core.js`, Funktion `parseBatchResponse`, letzte Zeile des `results.map`:
```js

```
Wenn ein LLM eine Übersetzung als korrekten deutschen Satz mit öffnendem Anführungszeichen liefert (`"Hallo, Bürger"` → Zitat im Text), wird das erste/letzte `"` abgeschnitten. Für Songs of Syx-Texte mit Game-Dialogue-Zitaten potenziell fehler­trächtig.

**M3 — `applyTranslations` — `replacements`-Parameter ist Dead**

Das vorberechnete `replacements`-Array aus `readFileJob` wird an die Funktion übergeben, aber ignoriert. Die Datei wird erneut mit Regex gescannt. Wasted computation + misleading Signatur.

**M4 — `fixGrammarBatch` catch-Pfad verliert `attemptCount`**
Datei: `src/translation-runtime.js`, Funktion `fixGrammarBatch`, catch-Block.
```js
} catch (e) {
    consecutiveGrammarFailures++;
```

`attemptCount` wird beim Error-Retry nicht inkrementiert → der `if (attemptCount >= 2) return` Guard greift nicht. Nur `consecutiveGrammarFailures >= 3` verhindert Endlosrekursion. Im Erfolgs-Retry (Terminologie-Verletzung) wird korrekt `attemptCount + 1` übergeben. Inkonsistenz.

**M5 — Gemini `responseFormat`-Struktur falsch**

Die Gemini API erwartet `generationConfig.responseMimeType` und `generationConfig.responseSchema` auf Top-Level, nicht unter `responseFormat.text`. Das Schema-Enforcement schlägt still fehl (API ignoriert unbekannte Felder), JSON-Antworten kommen trotzdem, aber ohne Schema-Enforcement — Batch-Size-Garantien entfallen.
